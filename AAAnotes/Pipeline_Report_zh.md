## 0. 执行摘要

`哨兵`项目实现了一个具备自主导航能力的 RoboMaster 哨兵机器人，由三个紧密耦合的 ROS 工作空间和一个**SLAM 算法栈**组成：

| 层级 | 工作空间 / 文件夹 | 作用 |
|---|---|---|
| 决策 | `decision_node/DecisionNode/` | 基于行为树 (BehaviorTree) 的战术大脑 + MCU 串口桥接 |
| 激光雷达驱动 | `Old_Nav/livox_ws/` | Livox MID360 ROS 驱动 (`livox_ros_driver2`) |
| 感知 / 导航 | `Old_Nav/sim_nav/` (catkin) | SLAM 算法 + **主要** 导航栈 `bot_sim` (D*-Lite) |
| 导航滤波实验 | `Old_Nav/Navigation-filter-test/` | 点云滤波研究软件包 |
| 标准化 | `Old_Nav/standarlization/` | 激光雷达-IMU 外参初始化 (`LiDAR_IMU_Init`) |
| 顶层启动 (Launch) | `Old_Nav/*.launch` + `run_3DNavUL_Test_with_decision.sh` | 组合启动入口 |

**激活的运行链路**（通过 `3DNavUL_Test_with_decision.launch` 和 systemd shell 包装器 `run_3DNavUL_Test_with_decision.sh` 验证）如下：

```
Livox MID360 驱动  ──►  imu_filter  ──►  hdl_localization (基于预构建 .pcd 地图的 NDT 定位)
                                      │
map_server (2D .yaml 地图)  ──────►  dstarlite (D*-Lite 规划器 + 控制器) ──► /cmd_vel
                                      │
strategy_node (BehaviorTree_CPP v3)  ─► /clicked_point  (目标点)
                                      │
mcu_communicator  ◄──── UART HK协议 (921600) ────►  STM32 (底盘 + 云台 + 裁判系统)
```

Point-LIO、hdl_graph_slam、FAST_LIO_LOCALIZATION、LiDAR_IMU_Init **可用但不在激活的启动链路中**；它们主要用于离线 (a) 构建全局 `.pcd` 地图，(b) 估计激光雷达-IMU 外参，以及 (c) 算法实验。

### 2.2 行为树 — `strategy_node`

- 库: **BehaviorTree.CPP v3** (`bt_factory.h`)。
- XML 配置: [config/strategy_tree.xml](../decision_node/DecisionNode/src/decision_node/config/strategy_tree.xml) (支持通过 Groot2 载入)。
- Tick: 私有参数 `tick_hz`， 默认 **20 Hz**。
- **43 个自定义节点**，分布为 7 大组 — 详见 [BehaviorTree_Nodes_Summary_简洁版.md](../decision_node/DecisionNode/BehaviorTree_Nodes_Summary_简洁版.md) 和 [README.md](../decision_node/DecisionNode/README.md)。

#### 节点分组

1. **占位 (Central Occupiable)** — `AccumulateCentralOccupiable`, `TriggerOnThreshold`, `ResetAccumulator`, `ResetCentralOccupiable`, `IsOccupyStatusFavorable`。
2. **追逐 (Chase)** — `InitChase`, `UpdateChaseTarget`, `PublishChaseGoal`, `ResetChase`。
3. **运动控制 (Motion)** — `CheckArrived`, `CheckAttacked`, `SetMotionFlag`, `PublishMotion` (运动标志 flags：0 停止 / 1 巡逻 / 2 闪避躲避 / 3 推进； 在 0–2 之间有 5 s冷却期)。
4. **恢复/补给 (Recover & Supply)** — `IsHealthFull`, `SetRecover`, `IsBulletFull`, `SetBulletUp`, `PublishRecover`, `PublishBulletUp`, `SetBulletNum` (DELTA差动模式/FIXED固定模式), `PublishBulletNum`。
5. **黑板状态更新 (State)** — `UpdateRefereeBB`, `UpdateNavigationBB`, `UpdateVisionBB` (占位桩), `UpdateTimersBB` (占位桩), `UpdateDerivedFlags` (计算派生状态，如 `derived.damage_2s`, `is_dead`, `is_in_danger`, `bullet_sufficient`)。
6. **条件判定 (Conditions)** — `IsGameStarted`, `IsSentryDead`, `IsSentryAlive`, `IsSentryInDanger`, `NotBulletSufficient`, `AggressiveAdvantage` (侵略优势), `IsAction`, `IntenseHarm` (结合迟滞区间作剧烈伤害判定)。
7. **行动 (Actions)** — `SetAction` (状态设置包含: INIT/INITPUSH/PUSH/OCCUPY/SUPPLY/RESPAWN/RADICAL/RADICAL1/WAITFOROP), `ClearGoal`, `Wait`, `SetGoalFromParams`, `SetGoalFromParamsCyclic`, `AdvanceCycleIndex`, `PublishGoalPoint`。

#### 顶层行为树结构 (ReactiveSequence):
```
1. UpdateBlackboard（黑板状态更新）
   UpdateRefereeBB → UpdateNavigationBB → UpdateVisionBB → UpdateTimersBB → UpdateDerivedFlags
2. DecideAction (ReactiveFallback) — 基于不同优先级决定对应行动：
   Dead (阵亡) → RESPAWN; Respawn+Alive (重生后存活) → SUPPLY; InDanger (遇险) → SUPPLY;
   Init+GameStart (比赛开局) → PUSH; PUSH+Arrived (推进到位后) → OCCUPY; AggressiveAdvantage (进攻优势下) → RADICAL; 默认项为 WAITFOROP。
3. ExecuteAction (ReactiveFallback) — 下发目标点：
   PUSH/OCCUPY → SetGoalFromParamsCyclic (轮询取点) ns=occupy point_count=2
   SUPPLY → ns=supply; RADICAL/RADICAL1 → ns=radical/radical1; WAITFOROP → ns=waitforop
   所有分支的终点均流向 PublishGoalPoint ，将目标通过话题发布到 `/clicked_point` 以驱动导航。
```

### 2.5 `mcu_communicator` — HK UART 协议网关

头文件: [include/decision_node/mcu_comm.hpp](../decision_node/DecisionNode/src/decision_node/include/decision_node/mcu_comm.hpp)
源文件: [src/mcu_communicator.cpp](../decision_node/DecisionNode/src/decision_node/src/mcu_communicator.cpp)
启动文件: [launch/mcu_communicator.launch](../decision_node/DecisionNode/src/decision_node/launch/mcu_communicator.launch)

**通信协议规范 (HK)**
- 串口波特配置: 921600 8N1。
- 帧头 (SOF): 固定为 `'H' 0x48, 'K' 0x4B`; 帧尾: 固定为 `'K' 0x4B, 'H' 0x48`。
- 头端 10 Bytes 解析: `sof[2], length(le16), packet_type, reserved, packet_seq, reserved2, header_crc8`。
- 包类型: `0x01` = 比赛场控裁判传回数据 (MCU → PC), `0x02` = 导航指令 (PC → MCU, **大小在代码内做 static_assert 限定为 21 字节**)。
- 尾端携带 CRC16，对字节 0 … (len − 4) 负责进行校验判断。

**导航控制帧结构 (PC→MCU) 封包 Payload**
`reserved0, at_place, vx (int16 mm/s), vy (int16 mm/s), wz (int16 0.01 rad/s)`
— 这是借由 `sendNavigationCommand()` 在定时器以率频 `nav_frequency=50` Hz 由 `navigation_timer_` 所统一发出驱动并投往命令口的封装，其数据通过订阅 `/cmd_vel` 获取输入量。

**系统输入侧的话题订阅关联**

| 话题Topic | 类型 | 用法 |
|---|---|---|
| `/cmd_vel` | `geometry_msgs/Twist` | vx/vy/wz 映射包装填装 → 填发至发往 MCU 平台的导航通信帧 |
| `/navigation` | `geometry_msgs/Vector3` | 一种通过另外一侧来下发直驱控制的可替代接口 |
| `/nav_received` | `std_msgs/UInt8` | ACK 应答包回馈记录监听 |
| `/dstar_status` | `std_msgs/Bool` | 获取并映射传给下端标记 `at_place` 状态量 |

**通讯系统向外广播产出的内容（供给 strategy_node 的 BT 黑板体系调用）:**
囊括一切由于 MCU 下行的所有裁判系统与底盘数据: `/referee/*`, `/robot/*`, `/enemy/*`, `/radar/*`，外加获取到并且由云台以及下部分底盘送交回来的实机俯仰航偏角信息: `/mcu/yaw_angle` 和 `/mcu/chassis_imu` (`std_msgs/Float32`)。

该底层链路通信服务实现了能够热重连启动的冗余设计，即便物理层设备 `/dev/ttyUSB0` 因为拔除或是重启重载出现通信丢失中断故障，系统能够识别异常尝试再次热挂接(极大适配 USB 的重连特征)。并且它的主体节点代码本身根据上下发往回馈之差记录并在自身系统本地内挂载上了专门核算比赛总分数差对 (`friendly_score_` / `enemy_score_`) 以及机器人自身死亡判断标志状态机逻辑模型。

### 3.2 图形/场景建图辅助处理功能链条体系 — `hdl_graph_slam`

总控系统文件及目录分布： `Old_Nav/sim_nav/src/hdl_graph_slam/` · 相关查阅用笔记与记载资料在 [AAAnotes/hdl_graph_slam.md](hdl_graph_slam.md)。

整架构里面下置组合了 4 项 Nodelets 结构：
- `prefiltering_nodelet` —  (利用预设策略通过体素降采以及按限定范围进行截选感兴趣ROI的方式预处理过滤削尖降低算力点阵消耗)
- `scan_matching_odometry_nodelet` — 作为它的整个处理数据获取源和处理前端估算产生端，可被选取是按调配去通过加载和跑 `ndt_omp` 计算解算器或者是 `fast_gicp` 系统来进行初探里程计预处理估算的活。
- `floor_detection_nodelet` — （辅助分析解算节点）：调用 RANSAC 原则框架来进行提拔辨认真拟寻找地面点位、测探平坦并做地面判定剔除补偿机制。
- `hdl_graph_slam_nodelet` — g2o（基于整体位姿参数作建图图处理后部的整体图形纠偏与后部修差与纠错融合项系统库系统，也涵盖包拢对于大面积图跑出来的闭环查验控制项），并在一定状态将附加了地面点地平线信息或者额外的引入其它系统例如带入到了IMU或者是甚至把 GPS 的量等一些强制外部信息进行强制融合补偿与收敛。）

**生成出输送系统与产物文件结果导出路径系统流：** `/hdl_graph_slam/map_points` → 是借于使用了伴行附属配置工具包（也就是写在一个由总发文件 [Old_Nav/hdl_graph_slam_mapping.launch](../Old_Nav/hdl_graph_slam_mapping.launch) 控制的一个按照固定在时常为每隔 10秒触发激活的一起的 Python 挂钟脚本 `save_map_on_update.py` 处自动不间断发讯并把获取输出下来的实时生成点阵文件结构导存进 `$(find hdl_graph_slam)/map/mapping_result.pcd` 此目的位。
